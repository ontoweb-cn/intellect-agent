multi profile routing implementation plan

For Hermes: Use subagent-driven-development skill to implement this plan task-by-task. Goal: Enable a single Gateway process to serve multiple profiles simultaneously via per-user/per-chat profile bindings, eliminating the need to restart the gateway to switch profiles. Architecture: Add a profile_bindings table to state.db, retrofit GatewayRunner to maintain per-profile agent caches, and inject profile resolution into _handle_message → _run_agent. Profile resolution follows priority: per-user binding > per-chat binding > gateway default. Tech Stack: Python 3.12+, SQLite (state.db, schema v11→v12 migration), asyncio

Overview

This plan modifies the Gateway to internally route incoming messages to the correct profile’s AIAgent, instead of binding the entire Gateway process to a single INTELLECT_HOME at startup. The existing single-profile path is preserved as a degraded fallback: when no bindings exist or multi_profile.enabled is falsy, the Gateway continues to use its own INTELLECT_HOME exactly as before.

Key design decisions

  1. Profile resolution lives in gateway/profile_router.py — a new dedicated module that queries state.db for bindings and merges per-profile configs. This keeps the routing concern separate from the existing 17,000-line gateway/run.py.

  2. AIAgent cache becomes dict[str, OrderedDict] — keyed by profile_name first, then session_key. The existing _AGENT_CACHE_MAX_SIZE and _AGENT_CACHE_IDLE_TTL apply per-profile, so a busy profile doesn’t evict agents from an idle one.

  3. Config merging follows deep_merge() — the per-profile config.yaml is loaded on first use and shallow-merged with the Gateway’s own config, with profile values taking precedence. This means a profile can override its model, system prompt, toolsets, and memory settings independently.

  4. State.db schema v12 adds one new table (profile_bindings) with zero migration risk — the migration is an additive CREATE TABLE IF NOT EXISTS.

What does NOT change

  • Gateway startup / shutdown lifecycle — same start_gateway(), same PID file, same lock scoping under INTELLECT_HOME.
  • SessionStore / SessionDB — sessions remain scoped to INTELLECT_HOME, not profile. This is intentional: the Gateway host is the session owner, and profiles are an agent-configuration concern.
  • Kanban dispatcher — _default_spawn() already passes -p <assignee> to subprocess invocations. No changes needed.
  • CLI / intellect chat — profile switching via -p <name> is unchanged.

Risk Assessment

RiskSeverityMitigation
INTELLECT_HOME env var leak between turnsHighfinally block in _run_agent ensures restoration even on exception
Agent cache growth (N profiles × M sessions)MediumPer-profile cache cap, configurable via MultiProfileConfig
Config merge surprises (profile overrides gateway)Low_deep_merge preserves all base keys; profile only overrides what it declares
Profile not found (deleted from disk)Lowget_profile_config returns empty dict on missing config.yaml, graceful degradation to base config
SQLite lock contention (frequent profile lookups)LowProfileRouter.resolve() opens a short-lived SessionDB() and closes it immediately; WAL mode ensures concurrent access

Edge Cases

  1. Profile deleted while Gateway is running — next message to that profile falls back to base config (graceful degradation). Logs a warning.
  2. Binding created while user is mid-conversation — takes effect on the next message, not the current turn.
  3. config.yaml has YAML syntax errors — yaml.safe_load exception is caught, returns empty dict, logs warning.
  4. Multi-profile disabled mid-session — ProfileRouter.enabled is read on every resolve() call. If disabled via hot-reload, subsequent messages use the default profile. Cached per-profile agents are not cleaned up until Gateway restart or idle TTL eviction.
  5. SessionDB unavailable — resolve() returns default_profile. The Gateway can still process messages with its own config as a single-profile fallback.
