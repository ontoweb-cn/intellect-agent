LLM+P vs KnowAgent — 深度对比

两条路线解决同一个问题（LLM 规划不可靠），但走了完全相反的路：LLM+P 让 LLM 完全不参与规划计算，KnowAgent 用结构化知识约束 LLM 的规划过程。

一、架构对比

LLM+P：LLM 不碰规划

┌────────────┐     ┌──────────────────┐     ┌────────────┐
│   用户 NL   │ ──→ │    LLM (翻译器)   │ ──→ │   PDDL     │
│ "整理积木"  │     │ NL → PDDL problem │     │ 形式化问题  │
└────────────┘     └──────────────────┘     └─────┬──────┘
                                                   │
                                          ┌────────▼──────┐
                                          │ Fast Downward │
                                          │  (Planner)    │
                                          │ 状态空间搜索   │
                                          └────────┬──────┘
                                                   │
┌────────────┐     ┌──────────────────┐     ┌─────▼──────┐
│  可执行计划 │ ←── │    LLM (翻译器)   │ ←── │  符号计划   │
│ 自然语言    │     │ PDDL plan → NL   │     │ [A,B,C,D]  │
└────────────┘     └──────────────────┘     └────────────┘

LLM 的两端是翻译，中间的计算完全交给 Classical Planner。LLM 不参与任何规划推理。

KnowAgent：LLM 仍在规划，但被约束

                    ┌─────────────────────┐
                    │   Action KB          │
                    │ ┌─────────────────┐  │
                    │ │ search:          │  │
                    │ │  pre: entity_type│  │
                    │ │  post: candidates│  │
                    │ │  next: [lookup]  │  │
                    │ ├─────────────────┤  │
                    │ │ lookup:          │  │
                    │ │  pre: searched   │  │
                    │ │  post: details   │  │
                    │ │  next: [pick]    │  │
                    │ └─────────────────┘  │
                    └──────────┬──────────┘
                               │ 过滤合法后继
                               ▼
┌────────────┐     ┌──────────────────────┐     ┌────────────┐
│   用户 NL   │ ──→ │    LLM (规划 + KB约束) │ ──→ │ TAO 轨迹   │
│ "找苹果捡起来"│     │ Thought→Action→Obs   │     │ 可执行序列  │
└────────────┘     └──────────────────────┘     └────────────┘

LLM 全程参与规划，但每一步都从 Action KB 获取合法候选项，在受限空间内做选择。

二、LLM+P 详细机制

核心思想

LLM 擅长语言理解和转换，但不擅长可靠搜索；Classical Planner 擅长在形式化状态空间中求解计划，但需要结构化输入——取长补短，各司其职。

PDDL：规划的”数学语言”

Planning Domain Definition Language，用谓词逻辑精确描述动作的前置条件与效果。

Domain PDDL（“世界的物理规则”）

Problem PDDL（“初始状态 + 目标”）

用户: "从冰箱里拿一瓶牛奶放到桌上"

LLM 翻译 →
(define (problem fetch-milk)
  (:domain household-tasks)
  (:objects
    fridge table - location
    milk - object)
  (:init
    (inside milk fridge)
    (not (container-open fridge))
    (handempty))
  (:goal
    (at milk table)))

三步流水线

Step 1: LLM 将 NL 翻译为 Problem PDDL

LLM 读入已知的 domain PDDL，将自然语言问题翻译为结构化的初始状态和目标状态。

Step 2: Classical Planner 搜索

Fast Downward 读取 domain + problem，在状态空间中执行 A* / 启发式搜索：

状态 S0: milk在冰箱里, 冰箱关着, 手空
  │
  ├── goto(fridge) → S1: 在冰箱前
  │     ├── open(fridge) → S2: 冰箱开着
  │     │     └── take-out(milk, fridge) → S3: 拿着牛奶
  │     │           └── goto(table) → S4: 在桌前
  │     │                 └── put-down(milk) → S5: 牛奶在桌上 ✓ 目标达成！
  │     │
  │     └── ... (其他分支被剪枝)

输出: [goto(fridge), open(fridge), take-out(milk,fridge), goto(table), put-down(milk)]

Step 3: LLM 翻译回自然语言

Planner 输出符号计划 → LLM 翻译 →
"1. 走到冰箱前
 2. 打开冰箱门
 3. 从冰箱里拿出牛奶
 4. 走到桌前
 5. 把牛奶放在桌上"

关键特性

Planner 保证找到的计划是 (1) 可执行的、(2) 最优的（最短步数/最小代价）、(3) 数学上正确的（Sound + Complete）。

三、KnowAgent 详细机制

核心思想

LLM 规划时犯错是因为不知道动作之间的约束关系。给它一张”动作地图”——显式编码每个动作的前置条件、后置效果、合法后继——LLM 在受限的合法空间内推理，显著减少 invalid action 和 misordered action。

Action KB 结构

TAO 生成过程

以”从冰箱拿牛奶放到桌上”为例

═══════════ Turn 1 ═══════════
Action KB 注入 → "初始状态：handempty=true。可用动作：[search, navigate_to]"

Thought: 先找到牛奶。冰箱应该在厨房。
Action: search("milk")
  ├── KB 检查: search 前置条件通过 ✓
  └── KB 记录: 合法后继 = [lookup, filter]
Observation: 找到 "milk_01" 在 fridge_01 中

═══════════ Turn 2 ═══════════
Action KB 注入 → "已搜索到 milk_01 ∈ fridge_01。合法动作：[lookup(milk_01), navigate_to(fridge_01)]"

Thought: 确认牛奶确实在冰箱里，然后过去。
Action: lookup("milk_01")
  ├── KB 检查: lookup 前置条件通过 ✓
  └── KB 记录: lookup 后可执行 pick
Observation: milk_01 是 500ml 全脂牛奶，在 fridge_01

═══════════ Turn 3 ═══════════
Action KB 注入 → "已确认 milk_01 在 fridge_01。
  ⚠ pick 前置条件：agent 必须在目标位置 && 容器必须打开"

Thought: 还不在冰箱前，冰箱也没开。先去冰箱那里。
Action: navigate_to("fridge_01")
  ├── KB 检查: 不在 fridge_01，navigate_to 合法 ✓
  └── KB 更新: 位置 = fridge_01
Observation: 已到达冰箱前

═══════════ Turn 4 ═══════════
Action KB 注入 → "位置=fridge_01。已知 fridge_01 是容器，container-open=false。
                   合法动作: [open(fridge_01)] ← 只有这一个"

Thought: 冰箱关着，先打开。
Action: open("fridge_01")
  ├── KB 检查: fridge_01 是容器 ✓
  └── KB 更新: container-open(fridge_01)=true
Observation: 冰箱已打开

═══════════ Turn 5 ═══════════
Action KB 注入 → "位置=fridge_01, holding=none, container-open=true。
                   合法动作: [pick(milk_01, fridge_01)]"

Thought: 拿出牛奶。
Action: pick("milk_01", "fridge_01")
  ├── KB 检查: 所有前置条件满足 ✓
  └── KB 更新: holding=milk_01, handempty=false
Observation: 拿到了牛奶

═══════════ Turn 6-7 ═══════════
→ navigate_to("table") → put_down("milk_01", "table")
→ ✓ 牛奶已在桌上，任务完成

Action KB 的三个介入点

  1. Planning 阶段：过滤非法候选项，LLM 只能从合法后继中选
  2. Generation 阶段：在 prompt 中注入当前可用 action 的约束描述
  3. Verification 阶段：检查生成的 action 序列是否满足全局约束

四、核心差异全景

维度KnowAgent (Action KB)LLM+P (Classical Planner)
规划者LLM (被 KB 约束)Classical Planner (Fast Downward)
LLM 角色每一步推理 + 选择纯翻译 (NL ↔ PDDL)
知识形式Action KB (前置/后置条件)Domain PDDL (STRIPS 谓词)
正确性概率性 (KB 减少但未消除错误)证明性 (Sound + Complete)
知识获取半自动 / LLM 辅助构建需要人类专家手写
实时约束每一步注入 KB一次性翻译后全部交给 Planner
适应范围开放域、难以形式化可形式化为 PDDL 的经典问题
出错模式KB 不完整 → 漏掉约束翻译错误 → Planner 做”精确的错事”

哲学分歧

KnowAgent: "LLM 规划不好是因为它不知道规则，
           给它规则就行——它仍然是最佳推理引擎。"

LLM+P:     "LLM 规划不好是因为它天生不适合搜索，
           别让它规划——让它做它擅长的翻译。"

五、致命弱点

KnowAgent

问题表现
KB 覆盖率Action KB 漏掉一个约束 = LLM 照样犯错。例如 KB 没写”拿冷冻物品前要戴手套”，LLM 会直接伸手。
状态爆炸真实环境的状态空间远超 KB 能编码。一个酒店预订 API 可能有 30+ 参数组合约束。
KB 构建成本每次增加新动作/新领域都要更新 KB。
无全局最优保证LLM 每步选合法的，但可能走入局部最优（贪心）。Planner 做全局搜索，保证最优。

LLM+P

问题表现
PDDL 表达力天花板许多真实问题写不成 PDDL。模糊目标、软约束无法形式化。
翻译质量是瓶颈LLM 把 “找最近的面包店” 翻译成错误的 PDDL 谓词和对象，Planner 给出精确但完全错误的计划。
Domain PDDL 必须预知每个新领域需要人类专家编写 domain 文件。
丢弃 LLM 世界知识Planner 只知道 PDDL 里的规则。LLM 知道”冰箱通常在厨房”、“牛奶会坏要冷藏”，这些常识被完全丢弃。

六、场景选择指南

场景推荐理由
物流调度、路径规划LLM+P经典 PDDL 领域，Planner 有最优解
API 编排（Dify/Coze workflow）KnowAgentAPI 参数约束复杂，但可以写成 KB
积木世界、棋盘游戏LLM+P天然适合 PDDL
客服对话流程KnowAgent对话状态难以形式化为 PDDL 谓词
机器人任务规划（结构化环境）LLM+P经典 Planning 领域
机器人任务规划（开放环境）KnowAgent物品、位置不固定，依赖 LLM 常识

决策树

问题可以精确形式化？
  ├── 是 → 动作空间有限、依赖关系清晰？
  │        ├── 是 → 用 LLM+P（Planner 保证最优）
  │        └── 否 → 用 KnowAgent（KB 约束 + LLM 常识补充）
  │
  └── 否 → 动作是自然语言、软约束、模糊目标？
           └── 用 KnowAgent（KB 约束 + LLM 灵活推理）

七、第三条路：LLM-Modulo

Critical Investigation (NeurIPS 2023) 提出的 LLM-Modulo 框架是两者的折中：

LLM-Modulo:
  LLM 生成候选计划
       ↓
  Verifier 检查可执行性  ← 类似 Action KB 的约束检查
       ↓ (如果失败)
  Planner 修复           ← 类似 LLM+P 的符号修复

vs KnowAgent: LLM-Modulo 是"先出计划再修"，KnowAgent 是"每一步约束着出"
vs LLM+P:     LLM-Modulo 中 LLM 仍然生成初始计划，LLM+P 中 LLM 完全不碰规划
方法LLM 参与度外部符号系统角色保证级别
LLM+P零（纯翻译）全部规划计算数学证明
KnowAgent高（约束下推理）提供约束概率性减少错误
LLM-Modulo中（生成候选）验证 + 修复最终计划可执行

八、趋势与融合方向

现实中最优解很可能是两者的混合：

  1. 用 Action KB 约束 LLM 生成候选计划：降低非法动作，保留 LLM 常识
  2. 用 Classical Planner 验证关键路径：对可形式化的子问题提供数学保证
  3. 用执行反馈闭环修正：类似 LLM-Modulo 的 Verifier 环节，用实际执行结果验证
