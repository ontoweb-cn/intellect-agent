一、细分结构化及时序的Agent记忆思路HINDSIGHT

来看一个有趣的工作，基于细分结构化记忆的Agent记忆思路 ，做的结构化长期记忆架构 ，核心就是细分了个网络记忆结构 （世界、经验、观点、观察，也就是world, experience, opinion, observation，然后将每个提取的事实做路由分类到这几个里面，所以可以理解是细化 ），工作在《HINDSIGHT IS 20/20: BUILDING AGENT 》， 地址在：https://hindsight.vectorize.io，https://github.com/vectorize-io/hindsight，报告地址在：https://arxiv.org/pdf/2512.12818，https://github.com/vectorize-io/hindsight/blob/main/hindsight-docs/static/img/retain-operation.webp，

重点看看记忆的构建和检索思路，找些思路，尤其是做的这个路由分类、这个时序上的检索机制

1、记忆的构建方式

这个操作会用大模型从输入中提取关键事实、时间信息、实体和关系 ；再通过标准化流程，把提取的数据转化为规范实体、时间序列、检索索引+元数据 ；这些结构化表示，会为后续Recall（回忆/检索）和Reflect（反思/推理）操作，做索引通路 。

这个流程也不闲再啰嗦，可以再看下：

RetainInput ，做记忆留存，输入可以是用户对话、文档、日志、多模态信息等原始数据；

step1.抽取

LLMFactExtraction ，用大模型对原始输入做结构化抽取，生成包含每个事实的叙述性文本、规范化的时间信息（包括时间范围）、参与者及其角色、指示目标网络的事实类型：

其中，将相对时间引用（如“上周”、“三月”）转换为绝对时间戳 (τs, τe)；

事实类型分类，根据陈述的性质分配 {world, experience, opinion, observation}，这很粗暴 。

step2：对齐、消歧

Entity Recognition & Resolution，实体识别 + 消歧，从事实中定位实体，并解决同名、别名问题，统一实体指代。

其中，实体抽取，识别PERSON、ORGANIZATION、LOCATION、PRODUCT、CONCEPT和 OTHER 等实体类型 。

消歧方面，通过结合字符串和名称相似度（例如，Levenshtein 距离）、与其他实体的共现模式以及提及的时间接近性，通过最大化相似度得分，这个很常用 。

Normalized Canonical Entities ，输出标准化的规范实体。

step3：建立关系边

包括两步：

一个是Create Memory Units（创建记忆单元 ），把前面的事实、实体、时间信息，封装成最小记忆单元，每个单元包含text 【原始/摘要化的事实文本】、embeddings 【文本的向量嵌入（用于语义检索）】、timestamps 【时间戳（用于时间维度检索）】、metadata 【元数据（如来源、事实类型、置信度等）】；

一个是Link Construction（关联构建） ，给记忆单元之间建立多维度关联，让记忆从孤立条目变成知识图谱，包括；

Entity Links 做实体关联、Temporal Links做时间关联，时间先后关系。

Semantic Links 【语义关联，比如AI算法和大模型、知识图谱的语义相似/从属关系】；

Causal Links 【因果关联，比如用户学习Agent技术→用户关注OpenClaw项目的因果关系】；

step4:记忆路由存储

抽取好后，存储，按记忆的类型，把记忆单元分发到不同的存储分区 ，做分类管理【为了效率，每个路由类型对应一个独立的存储分区，方便按类型快速检索】，具体的包括多个，World世界知识/客观事实+Experience-个人/ Agent的经历、交互历史+Opinion-观点、偏好、态度+Observation-观察、感知到的信息 。

2、记忆检索方式

检索方式还是很常规的Multi-Strategy Retrieval (TEMPR)，这种典型的多路召回策略，RAG中常用，也可以回顾下RAG的做法。

语义检索 ，用于概念相似性、同义转述场景；

关键词检索（BM25） ，用于人名、专业术语、精确匹配场景；

图谱检索 ，用于关联实体、间接关联推理场景；

时序检索 ，用于“去年春天”“六月”、时间范围类查询，这个可以展开看看【会用到】

当查询中检测到时间约束时 ，调用一个由混合时间解析器支持的时间图检索，具体的首先运行一个基于规则的分析器 ，使用两个现成的多语言日期解析库，将显式和相对表达式（例如，“昨天”、“上个周末”或“2024年6月”）归一化为一个日期范围 。

这种启发式路径以低延迟处理了大部分查询。对于无法通过启发式方法解决的查询，回退到一个轻量级的序列到序列模型（此处使用google/~an-t5-small），将剩余的时间表达式转换为具体的日期范围[τstart,τend]。随后，将其与记忆的出现区间进行匹配。

最后，采用 cross-encoder/ms-marco-MiniLM-L-6-v2 模型，模型会联合编码查询和每个候选记忆项，并输出相关性得分。

二、面向AI编码助手的知识图谱构建插件graphify

来看一个工作，面向AI编码助手的知识图谱构建技能，也可以叫做插件，其打的点还是，基于图的存储，降低token消耗 。

工作在：https://github.com/safishamsi/graphify/blob/v3/README.md，这类工具其实很多了，zep、graphiti等有好多，但是再熟悉下也行，看看怎么做，有哪些组件，最终成型是啥。

其slogan是：在ClaudeCode、Codex、OpenCode或OpenClaw中输入/graphify，读取你的文件、构建知识图谱，口号是直接丢进去代码、PDF、Markdown、截图、流程图，用Claudevision从这些内容中提取概念和关系，然后搞成图谱。

1、工作原理

因为面向的数据不一样，所以需要采用不同的处理机制，如下：

对于**.py .ts .js .go .rs .java .c .cpp .rb .cs .kt .scala .php .swift .lua格式的代码（Code），通过 tree-sitter生成抽象语法树（AST）+调用关系图+文档字符串/注释中的设计逻辑，这是一款开源的语法解析工具，可生成 语言代码的抽象语法树 （AST，代码语法结构的树形表示，用于精准提取类、函数、call-graph（调用关系图，记录代码中函数/方法之间调用关系的图结构，用于梳理代码执行链路）、docstring/commentrationale（代码中的文档字符串和注释，用于提取代码的设计思路、实现逻辑代码结构信息）这个是做 code graph，以及做chunking的常用操作 ，代码在：https://github.com/safishamsi/graphify/blob/v3/graphify/extract.py

对于.md，.txt，.rst格式的文档（Docs） ，通过Claude提取概念、关联关系与设计逻辑；

对于pdf格式的论文（Papers） ，进行citation mining （引用关系挖掘，从论文中提取参考文献、引用链路，梳理学术脉络）+概念提取；对于.png，.jpg，.webp，.gif格式的图像，使用Claude视觉能力做提取，可识别图像中的文字、图表、截图等内容。

然后，提取其中的概念、关联关系与设计逻辑的细节 是，所有关联关系均标注成三个类型，包括EXTRACTED （提取，直接从源码中获取）、INFERRED（推断，合理推导结果，附带置信度评分）、AMBIGUOUS （歧义，标记待人工复核），这里的设计逻辑是：可清晰区分哪些信息是直接提取所得，哪些为推导猜测 。

然后，上述结果会合并为一张NetworkX图 ，通过莱顿社区发现算法 进行聚类，最终输出为交互式HTML文件、可查询的JSON数据以及review报告。
