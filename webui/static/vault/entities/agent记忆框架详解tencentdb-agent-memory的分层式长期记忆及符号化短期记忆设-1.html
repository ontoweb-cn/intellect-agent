今天是2026年5月24日 ，星期日，北京，天气晴

来看Agent记忆方面的进展，我们之前已经讲过很多了，尤其是基于图的Agent记忆、基于树的、基于层次的记忆方案等 ，都在解决上下文超载问题、个性化问题 。

最近有个方案很火，TencentDB-Agent-Memory，基于上下文卸载与结构化图表示的短期记忆压缩方案 ，通过将完整信息卸载至外部文件系统，同时利用 Mermaid将任务执行过程转化为可导航的结构化记忆 ，这个Mermaid的方案有意思，所以展开来看看。

官方的地址在《TencentDB Agent Memory 》(https://github.com/Tencent/TencentDB-Agent-Memory)，对应的解读在：https://deepwiki.com/Tencent/TencentDB-Agent-Memory，https://zread.ai/Tencent/TencentDB-Agent-Memory，官方也有一些解读，在：https://mp.weixin.qq.com/s/mK5WoNgJSPSpdmsUSSDqPw，又如：https://mp.weixin.qq.com/s/MSXKfefrqM31q-7WXIqCEg，解读基于此开展。

先说具体结论：

其一，TencentDB Agent Memory=符号化短期记忆+分层式长期记忆 ，也就是用于渐进式分层长期记忆，以及用于任务内上下文压缩的符号化短期记忆，分开处理。

其二，在实际处理时候，这些各层也是互相协同的。分成两个阶段：

提示词构建前（handleBeforeRecall）阶段 ：召回流程搜索L1记忆（通过混合向量+关键词搜索），加载L3人设，并组装场景导航。它们作为prependContext（每轮的L1记忆）和appendSystemContext（稳定的人设+场景导航+工具指南）被注入到提示词中。

轮次结束后（handleTurnCommitted）阶段 ：将新消息写入L0，当累积了足够的新消息时触发L1提取，并根据各自的触发条件调度L2场景聚合和L3人设生成，与此同时，卸载流水线将工具结果压缩为Mermaid符号。

与此对应的，就是是目前很流行的渐进式披露，Agent 在上下文中关注紧凑的高层结构（人设、场景导航、Mermaid 画布），仅在需要时向下钻取至原始证据。长期记忆上，钻取路径始终是确定性的—从一个人设特征，到促成它的场景，到这些场景内的 L1 记忆，再到产生这些记忆的 L0 消息；而对于短期任务，则是从一个 Mermaid node_id 到完整的卸载日志 。

基于这些，本文分别看分层式长期记忆和符号化符号化短期记忆具体怎么做的，里面的mermaid是怎么做的，并最后做个总结。

一、TencentDB Agent Memory的分层式长期记忆

长期记忆指的是历时对话，底层是原始对话，顶层是提炼后的用户画像。每一层对下层进行提炼与压缩 ，Agent在推理时只需关注顶层的抽象即可，当需要细节时，可以沿着确定性的溯源链向下钻取。

因此，实现方式是建立L0 Conversation（原始对话）→L1 Atom（结构化事实）→L2 Scenario（场景块）→L3 Persona（用户画像） 的层级结构，平时靠高层Persona把握用户偏好，需要考证细节时再检索底层。

实现流程也是一个流水线：从L0对话捕获→L1提取与去重→L2场景聚合→L3人设生成 。

可以细分看下：

1、L0 Conversation（原始对话）

将每条用户和助手的消息原样记录到每日的 JSONL 文件中（conversations/YYYY-MM-DD.jsonl）。具体实现上，不进行摘要、不过滤，只是忠实地捕获，每条消息都携带着唯一的 id、sessionKey 和时间戳 。

2、L1 Atom（结构化事实） ，在L0的基础上，使用LLM提取出原子记忆

每条记忆被划分为三种类型之一，并被赋予一个优先级分数。

特别的，提取之后使用批量去重步骤，使用向量相似度召回，将新记忆与已有记忆进行比较，去重引擎为每条记忆决定以下四种操作之一：store（新事实）、update（替换过时事实）、merge（合并重叠事实）或 skip（已知事实） ，这能防止存储空间因冗余条目而无限制地增长。

结果被持久化到两个互补的存储中：仅追加的 JSONL 文件 （records/YYYY-MM-DD.jsonl）作为持久可靠的数据源，以及向量数据库（SQLite + sqlite-vec 或 Tencent Cloud VectorDB ）作为主检索引擎。每条记忆记录都带有 source_message_ids，将其链接回生成它的 L0 消息。

3、L2 Scenario（场景块） ，这一步的每个场景块都是一个Markdown 文件

包含一个META分隔头部，其后是叙事内容。META部分携带索引元数据——创建时间戳、最后更新时间戳、简洁的摘要和热度分数 。

具体实现上，使用一个带有文件工具的 LLM Agent-SceneExtractor，自主读取现有的场景文件(文件工具（read、write、edit）自主读写 .md 文件)，决定是CREATE（创建）一个新场景、UPDATE（更新）现有场景，还是 MERGE（合并）相似场景——然后直接写入 Markdown 文件 。当场景数量接近配置的最大值（默认：15）时，LLM被指示在创建前先进行合并，以保持场景空间的可用性。

二、短期记忆（上下文卸载/任务）

这里的短期记忆是在漫长的任务会话中，冗长的工具输出（搜索结果、代码清单、错误追踪）消耗了上下文窗口的绝大部分，所以，解决方式，就是上下文卸载和任务分层 ，通过一套独立于长期记忆管道的三阶段压缩流水线来实现 ，分成底层markdown文件、中层json条目、顶层mermaid流程图 三个层次。

在具体执行时，Agent在上下文中仅需关注顶层结构，遇错时再沿着node_id下钻到底层查证。

这里又可以进一步拆分出来L1.5如下：

具体的：

Step1. L1 ：工具输出 → 摘要 OffloadEntry + 原文存入 refs/*.md;

Step2. L1.5 ：判定长任务边界，激活 MMD 文件;

Step3. L2 ：后台生成/增量更新 Mermaid 流程图;

Step4. 注入器 ：把紧凑的 Mermaid 图注入上下文，替代冗长的工具历史;

Step5. L3 回收 ：当上下文继续膨胀时，轻度 ：用摘要替换工具结果（原文在 refs/ 中无损保留）；激进 ：删除最早的消息，但注入历史 MMD 图作为补偿（mmd-injector.ts#L294-L301）； 紧急 ：更激进截断.

这里的关键在于：每一步压缩都有回溯路径 。从 MMD 的 node_id → offload.jsonl 中的条目 → refs/*.md 中的完整原文。压缩不是失忆，而是渐进式披露。

这里要展开去讲几个点。

1、Mermaid流程图做任务图的逻辑

短期上下文卸载采用Mermaid流程图作为符号压缩层，这一步解决的是将庞大的工具调用历史转化为紧凑、语义丰富的任务图 。

关于这块，文章（https://mp.weixin.qq.com/s/MSXKfefrqM31q-7WXIqCEg ）做了个较好的解读， Mermaid有几个很适合 Agent 的特点，如下：可以表达节点、边、状态、依赖关系，节点可以绑定 node_id、summary、result_ref。

此外，重点尝试了两种格式：Flowchart 和 StateDiagram ，结论就是Flowchart 相比 StateDiagram 效果约提升 15%，在长任务上下文卸载场景中，Agent 更需要一张表达复杂依赖和多路径探索且足够自由的任务地图，StateDiagram 更适合描述有严格生命周期的对象，Flowchart 更适合描述 Agent 高度自由的探索式执行过程。

调用本地LLM，输入三样东西：现有 MMD 内容（带行号注释，方便精确引用）、新的 OffloadEntry 列表、近期对话历史，

LLM输出Mermaid图中每个节点有如下结构，如：001-N1["summary: 查询数据库状态<br/>status: done"] 001-N2["summary: 执行迁移脚本<br/>status: doing"]，node_id（如 001-N1）：唯一标识，可映射回 refs/*.md 中的原始工具输出，status：done / doing / todo，告诉 Agent 任务进展，MMD 文件头部还有 元数据 。

在形式化上，底层保留原始、厚重的工具调用结果（refs/*.md），中层抽取步骤摘要（jsonl），高层则浓缩为一张极度轻量的Mermaid任务画布。每个Mermaid图表均以.mmd文件的形式持久化存储在Agent的mmds/目录下，文件遵循严格的结构：

由分隔符包围的JSON元数据头部，其后是flowchartTD主体，主体中的节点均带有结构化标签。

其中，JSON元数据头部 编码了任务的身份标识与进度，对应【严格时间戳与元数据规则】：

1)顶部元数据（必填） ：（updatedTime为node中的最新时间）。

2)节点内时间 ：如果合并了多个新条目，节点内的 Timestamp 必须取其中最新的 ISO 时间。

此外也有【严格的工程底线】 ，如下：

1)节点标准格式 ：NodeID["阶段名: 宏观动作简述<br/>status: done|doing|paused|blocked <br/>summary: 核心结论摘要<br/>Timestamp: ISO8601"]

对应4种状态：

其中，blocked 状态是系统的防重复机制：通过将死胡同显式记录为墓碑节点，LLM 避免了浪费 token 去重新尝试已经失败的方法，L2 提示词将其称为”认知墓碑” (cognitive tombstones)。

3、mermaid的注入形式

当然，生成Mermaid图表还不够，图表必须被注入到 LLM 的消息数组中的正确位置，才能能充当压缩上下文，具体插入到上下文的原则是：

找到最新的user消息位置，如果它在对话后半段→插入到它之后，如果在前半段→找到尾部工具循环块的起始位置（距尾部≤30条），关键保护：不会拆散assistant(tool_use)→tool_result的配对，这样MMD恰好坐在”用户问题”和”进行中的工具循环”之间，Agent先看到任务概览图，再看具体工具操作 。

4、mermaid画布的执行意义

因此，在这个mermaid画布搭建完之后，可以看具体用法，这里援引https://mp.weixin.qq.com/s/MSXKfefrqM31q-7WXIqCEg 的论述：

第一层是鸟瞰/Overview 。Agent先看到的是任务级别的概览：之前做过哪些任务，哪些已经完成，哪些还在进行，当前用户的问题可能和哪一个历史任务有关。这一层不追求细节，而是帮助Agent快速判断方向；

第二层是聚焦 。当Agent判断某个任务相关时，它再打开对应的任务画布，查看这个任务内部的结构。此时，Agent看到的不是一长串对话历史，而是一张经过组织的任务地图：哪些步骤已经完成，哪些节点还在进行，哪些信息汇聚成了当前结论；

第三层是下钻/细节补充 。当画布上的节点信息仍然不够时，Agent才进一步追溯到更细的摘要或原始材料。也就是说，原文不是一直常驻上下文，而是在真正需要确认细节时才被重新展开。

三、最后的总结-优劣势

总的来说，TencentDBAgentMemory是一个面向AIAgent的分层记忆系统，核心解决两个问题：

长期记忆方面 ，通过跨会话记住用户偏好、项目上下文、SOP，通过L0→L1→L2→L3的渐进式蒸馏，每层由LLM驱动提取；

短期记忆方面 ，单次长任务中上下文窗口不够用，通过Mermaid符号压缩+外部文件卸载+三级回收机制，每一层压缩都保留确定性回溯路径（node_id→refs/*.md），区别于粗暴摘要，这种设计让Agent在上下文中只保留高层结构（人设、场景导航、Mermaid画布），需要细节时下钻。比扁平向量库更合适些。

其中有两个有趣的设计。一个是回溯性 ，这是核心保障，这意味着压缩不是有损摘要，是可逆的。此外，用流程图替代冗长工具日志，token占用降一个数量级，同时Mermaid是LLM天然能理解的结构化格式 ，增量更新机制（指纹比对），避免了每次全量重写。

但是，也有一些不足，也看下：

其一，配置复杂度过高，光是OffloadConfig就有20+个参数，真正用好这套系统需要理解大量调优参数 （l2NullThreshold、mmdMaxTokenRatio、aggressiveCompressRatio……），对普通用户门槛很高。

其二，LLM调用成本和延迟不可忽视，每个管道阶段 （L1提取、L1.5边界判定、L2Mermaid生成、L2场景聚合、L3人设生成）都需要LLM调用。一次长任务会话可能触发数十次LLM调用。延迟和成本是真实存在的，尤其是在高并发场景下。

其三，L2生成的Mermaid图完全依赖LLM输出 。如果LLM产生格式错误、node_id冲突、或者语义遗漏，系统虽然有backfillNodeIds做回填，但没有对Mermaid严格语义正确性的校验，可能出出错，导致Agent不稳定。

所以，适合自己的，才是最好的，东西设计的越复杂，其实不可控因素越多 。
