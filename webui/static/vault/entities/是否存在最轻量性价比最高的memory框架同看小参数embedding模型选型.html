二、是否存在最轻量、性价比最高的Memory框架？

昨天社区谈到Agent记忆的这个问题，起因是有人提问：目前最轻量、性价比最高的 Memory方案是什么。

从答案的角度来看，这个问题其实并不太成立，提这个问题，其实需要回答什么叫记忆，可以从https://memu.pro/blog/memu-best-practice中找到一个更具像化的理解。

Short-term Memory，短期记忆 ，机制流程如下：

其遵循一个用于管理对话上下文的直截了当的结构，输入系统提示+聊天记录（上下文，按顺序组织）+用户查询，输出为模型响应 。

但这个聊天记录，容易爆炸，所以会控长度，前面多少轮。

长期记忆Long-term Memory ，则更多一些，如下：

使用一种更复杂的结构，结合了静态和动态记忆，输入的是系统提示+静态记忆+聊天记录（短期内存）+用户查询+检索到的相关记忆 。

重点看其中多出来的部分。

从使用场景上看，其主要用于跨越多个会话，当用户在几个小时/天/周后接着提问，对于持久的用户偏好和历史互动以及当上下文超过实际限制（ >10万token，影响推理速度时），特点就是引入检索，召回相关的进来。

但是，要说最轻量的方案，就是直接控制上下文轮次，对于tn时刻的prompt，往前回溯 k个时刻，比如tn-k到tn，里面的q或者a，或者qa，作为一个新的prompt，但这个会很爆炸，超长。方案是把k打小，或者做q，a摘要 。

但这相关性不好，因为轮次很多，以及用户的i图来回变化，噪声很多。所以，进而会直接引入rag，把轮次信息存入向量库或者es，做召回，控制topk ，这就是目前最轻量，最暴力的方案。

但，这个会带来一个问题，因为上了rag，就涉及到排序问题，所以，memo0也好，Graphti也罢，就细化这个记忆，方案是结构化，抽取一些meta信息，时间信息，个性化信息，使得更好计算相关性 。

例如，memu的做法，就是对记忆进行分类/抽取，做成profile, reminder，event这些，类似于图谱的方式，搞成hobby, skill这种，然后拉边。

然后呢，又有一个新问题，就是记忆的动态性，因为会有话题切换，轮次跳跃问题，比如前面说喜欢春天，现在又说喜欢秋天 。

为了解决这个问题，所以就搞update ，比如momo0(http://arxiv.org/pdf/2504.19413)里面的这种，进行curd操作做更新维护，这个是很好的思路，这个在《也看Mem0^g引入Graph做Agent Memory管理实现原理及医药领域RAG技术总结》（https://mp.weixin.qq.com/s/eLyum3OBkiRMiKOYbKoxmg）中有过介绍。

首先是不带Graph的方案，Mem0 ，实现逻辑是通过专用模块从对话中提取、评估和管理重要信息，系统的完整流程包括提取和更新两个阶段。其中：在提取阶段，当摄入一对新消息时，系统利用对话摘要和最近的消息序列作为上下文，通过LLM提取一组显著的记忆 。

系统首先使用向量嵌入从数据库检索前s个语义上相似的记忆。这些检索到的记忆与候选事实一起通过称之为“工具调用”的函数调用接口呈现给大模型，LLM本身决定执行四种不同操作中的哪一种：当不存在语义上等效的记忆时，使用ADD创建新记忆；使用UPDATE用补充信息增强现有记忆；使用DELETE移除与新信息相矛盾的记忆；当候选事实不需要修改知识库时使用NOOP 。

然后，在检索阶段，执行Retrieve relevant memories- >Generate Assistant response->Create new memories from the conversation三个步骤，也是一个搜图的过程：

顺着Graph用来做这个记忆管理，那么Zep: A Temporal Knowledge Graph Architecture for Agent Memor y（https://github.com/getzep/graphiti），这个也是用图来做记忆存储的，构建和查询“时间感知知识图谱”，这块的细节在《**GraphRAG如何用于Agent做memory记忆管理？Graphiti实现解读及相关技术进展回顾** 》（https://mp.weixin.qq.com/s/S8BG_XIQ3FIYKNq_RsC0AQ）中有讲述，核心就是面向用户交互上下文，创建和查询随时间演化的知识图谱。

但是，我们发现，一旦加入到抽取、细粒度这些操作，就会造成一个很明显的问题，引入的步骤太多，抽取也有准确性问题，也有时延问题，定位也不好做 。

所以，综上，所谓最轻量，最具性价比的问题其实不成立 。

因为，最的问题，随评估对象而动态变化，随你的需求，准确性，复杂度而变？我们只能说，有一个对应的进阶方案。比如从纯控制prompt，到rag，到memo0做结构化curd，再到Graphmemo也就是Graphti这些，越复杂，越乌托邦，变量越多，越不可控。

而我们又总会去看榜单，例如，https://github.com/NevaMind-AI/memU/中测出来的 Locomo dataset结果，

这些都是一些主流的方式，但是需要考虑到时间、复杂度这些才有意义，社区成员测给过，memu检索一次需要1s，在对话场景并不适用。

因此，落地过程中，我们应该从最小化开始，找技术的祖宗，找最关键的那个开始，不假思索的找捷径，结局是误了自己。

技术不存在“最”，只有更合适
