Workflow Generation with LLMs

大模型生成、编排和优化多步骤工作流的能力。涵盖从自然语言描述到可执行 DAG 的完整 pipeline。

问题定义

LLM Workflow Generation 的核心挑战：给定一个用自然语言描述的高层目标，模型需要生成一组有序、可执行的子任务（actions/nodes），并正确处理它们之间的依赖关系（DAG 拓扑）。这比单纯的文字生成或单步工具调用困难得多：

  1. 结构约束：生成的子任务必须形成合法的有向无环图（DAG），节点之间不能有循环依赖
  2. 动作可行性：每个节点必须对应真实可调用的 API/工具，参数类型必须匹配
  3. 可执行性：最终生成的工作流必须能在目标平台（Dify、Coze、Apple Shortcuts 等）上实际运行
  4. 效率优化：在满足正确性的前提下，最大化并行度、最小化总延迟

主要技术路线

1. 搜索式：将 Workflow 优化建模为搜索问题

代表工作：aflow（ICLR 2025）

AFlow 的核心创新是将 workflow optimization 视为搜索问题，使用 MCTS（蒙特卡洛树搜索） 自动发现最优 agentic workflow。它提供一组 operators（Generate, Review, Revise, Ensemble, Test, Programmer）， 每个 operator 由 LLM 实现，通过执行反馈评价候选 workflow。

核心思想：手工设计 workflow 成本高、不通用 → 用搜索自动发现最优结构

2. 知识图谱增强：利用领域 KG 指导 Workflow 生成

代表工作：opus（arXiv 2024）、knowagent（NAACL 2025 Findings）

Opus 引入 Work Knowledge Graph (WKG) 显式编码业务领域知识：

  1. 将客户输入→目标输出→业务上下文组织为 workflow intention
  2. 从 WKG 中检索相关流程知识指导 Large Work Model 生成候选 workflow
  3. 多候选合成为 Workflow Graph，按成本/时间/质量约束做路径优化

KnowAgent 则构造 action knowledge base，告诉模型哪些动作的前置/后置条件关系， 从而在生成 Thought-Action-Observation 轨迹时减少 invalid action 和 misordered action。

核心思想：通用 LLM 缺乏领域-specific 的流程知识 → KG 提供结构化先验

3. 编译式：LLM 生成 DAG，调度器并行执行

代表工作：llmcompiler（ICML 2024）

LLMCompiler 打破传统 ReAct 的串行瓶颈：

  1. 模型先生成 function-calling task DAG，标识任务间依赖
  2. 调度器并行运行无依赖的函数调用
  3. 显著降低延迟和成本，部分任务准确率也提升

与 AFlow 的区别：LLMCompiler 的 DAG 由 LLM 直接生成（非搜索），重点是并行调度。

核心思想：ReAct 的”思考→行动→观察”循环造成不必要的串行 → 先规划 DAG 再并行执行

4. 翻译-规划：LLM 翻译问题 + 经典规划器求解

代表工作：llm-p（arXiv 2023）

LLM+P 采用分工策略：

  1. LLM 负责将自然语言问题翻译为 PDDL 形式化问题
  2. Classical Planner 在形式化状态空间中搜索最优计划
  3. LLM 再将形式化计划翻译回自然语言

核心思想：LLM 擅长语言理解但不擅长可靠搜索 → 把规划计算交给专业规划器

5. 微调式：用大规模数据训练模型学会 Workflow 编排

代表工作：workflowllm（arXiv 2024）

WorkflowLLM 不依赖外部规划器或 KG：

  1. 从 Apple Shortcuts + RoutineHub 收集大规模真实 workflow 数据 → WorkflowBench 数据集
  2. 将 workflow 转写为 Python 风格代码，补充任务描述、计划和 API 信息
  3. 微调模型，使模型内化 workflow 编排能力

核心思想：通用模型未见过真实 workflow 的复杂性 → 用真实数据微调

评测维度

维度关注点代表 Benchmark
拓扑正确性DAG 依赖关系是否合法worfbench (ICLR 2025)
可执行性能否在平台上实际运行chat2workflow (arXiv 2026)
动作准确性调用的 API/工具是否正确WorkflowBench (WorkflowLLM)
任务完成率最终是否满足用户需求Chat2Workflow 的端到端测试
效率/并行度延迟、成本、并行利用率LLMCompiler 的 ParallelQA

开放问题

  1. 跨平台泛化：当前方法多针对特定平台（Shortcuts/Dify/Coze），缺乏统一的 workflow DSL 标准
  2. 动态适应：Workflow 执行中如何根据中间结果调整后续步骤（closed-loop）
  3. 复杂约束：如何表达和满足时间、成本、安全等多维约束
  4. 人机协作：如何让人类在 workflow generation 过程中提供反馈和修正
  5. 长程依赖：Action 序列超过 50+ 步时，当前方法正确率急剧下降
